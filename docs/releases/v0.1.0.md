# Release v0.1.0

Date: 2026-05-20
Commit: see the `v0.1.0` tag target
Tag: `v0.1.0`

## Summary

`v0.1.0` is the first production-readiness release for Production RAG
Assistant. It contains a complete local production-style RAG backend with
workspace isolation, document management, chat sessions, streaming responses,
OpenAI-compatible providers, deterministic evals, observability, async export
jobs, deployment documentation, configuration preflight, secret manager mapping,
and API key role enforcement.

## Verified Provider Mode

The release supports both:

- default fake-provider mode for local development and CI
- OpenAI-compatible embedding, generation, query rewrite, and reranking when
  `OPENAI_API_KEY` is configured

The release gate uses fake providers by default so CI and local checks do not
require paid provider access.

## Included Capabilities

- FastAPI backend with `/chat`, `/chat/stream`, documents, workspaces, chat
  sessions, export jobs, health, metrics, and static web UI routes.
- Postgres and pgvector schema managed by Alembic migrations.
- Markdown ingestion, chunking, content hashing, fake embeddings, OpenAI
  embeddings, and reindexing.
- Hybrid retrieval with vector retrieval, sparse retrieval, metadata filters,
  reciprocal rank fusion, optional query rewrite, optional session history, and
  optional OpenAI listwise reranking.
- Fake generator and OpenAI Responses API generator with streaming support.
- Citation validation, refusal guards, structured provider errors, token
  metrics, latency metrics, and cost estimates.
- Workspace registry, workspace archive/restore, write guards for archived
  workspaces, bulk workspace operations, and operation audit logs.
- API key workspace access control through `API_KEY_WORKSPACE_ACCESS`.
- API key role enforcement through `API_KEY_ROLES` with `admin`, `operator`,
  and `viewer` roles.
- Async chat log export jobs with worker loop, downloads, retry, stale running
  job recovery, and export file retention cleanup.
- Production-style Dockerfile and Compose stack with API, migrate, Postgres,
  and export worker services.
- Configuration guide, deployment runbook, secret manager mapping, observability
  docs, eval trends docs, project handoff, and release checklist.

## Database Migrations

## Security Notes

- `API_KEY_ROLES` is a secret because it repeats API key values on the left
  side of the mapping.
- Production deployments should set `API_KEY_ROLES` explicitly. If it is empty,
  the application keeps backward-compatible `admin` access for all configured
  API keys.
- Production deployments should scope `API_KEY_WORKSPACE_ACCESS` explicitly.
- Run `uv run python -m backend.app.core.config_check --production` before
  promotion.
- Do not print expanded environment values when real secrets are present.

## Known Limitations

- API key roles are a production-friendly foundation, not a full user identity
  system. A complete user/organization model can be added later.
- In-process rate limiting is suitable for local or single-instance deployment.
  Multi-replica deployments should use Redis, an API gateway, or a reverse
  proxy layer.
- The web UI is intentionally minimal and served by FastAPI.
- Real provider quality and cost depend on the configured OpenAI-compatible
  models and price table.
